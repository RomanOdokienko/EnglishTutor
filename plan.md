# План развития — English Tutor

Это активный roadmap (июль 2026). Исторический лог MVP — [tasks.md](tasks.md),
архитектура — [Architecture.md](Architecture.md), справочники — [docs/](docs/README.md).
Задачи рассчитаны на итеративную работу: каждую можно взять в отдельной сессии,
контекст внутри задачи самодостаточен.

## Куда движемся (North Star)

Инструмент навсегда для двоих (Roman + Andrey), еженедельные удалённые звонки.
Цель — не «статистика после звонка», а замкнутая учебная петля:

> **план перед звонком → звонок с записью в одну кнопку → разбор → конкретные
> ошибки помечены как фокус → на следующем звонке видно, что закрыто.**

Ретроспективная половина построена (запись → AssemblyAI multichannel →
детерминированные derived-метрики v1 → страницы Session/Progress). Не построена
проактивная половина: брифинг, фокус-ошибки, закрытие. Это этап 2.

Сознательно отложено: Telegram-доставка (пока ок заходить на сайт), dark mode,
mobile, переименование highlights.html→session.html, полноценная приватность
(Cloudflare Access) — не брать эти задачи, пока не закрыты этапы 1–2.

## Целевая топология (решение принято, июль 2026)

Три среды, у каждой своя роль:

| Среда | Роль |
| --- | --- |
| **Railway** (уже задеплоен, авто-деплой) | Живой бэкенд для ОБОИХ пользователей: приём записи, транскрибация, анализ, хранение sessions/data/out, отдача свежих JSON. Ключи OpenAI/AssemblyAI живут только здесь. |
| **Vercel** (уже задеплоен, авто-деплой из git) | UI: статика из `out/web`. Страницы ходят за данными и API на Railway (api_base). |
| **Локалка Романа** | Только разработка. Не участвует в боевом сценарии. |

Боевой сценарий: Роман открывает Vercel-страницу Record во время звонка →
кнопка → стоп → браузер шлёт аудио напрямую на Railway → Railway транскрибирует
и анализирует → обе страницы (у Романа и у брата) читают свежие JSON с Railway.
Git-пуш для появления данных у брата НЕ нужен (git остаётся источником кода и
бэкапом данных, см. задачу 1.4).

Ключевой известный риск Railway: **файловая система эфемерна** — всё, что
сервер записал (sessions/, out/, data/), исчезает при редеплое, если не
подключён Volume. Пока задача 1.1 не сделана, боевые записи на Railway делать
нельзя.

## Этап 1 — боевая петля: «нажал запись → статистика у обоих»

### 1.1 Персистентное хранилище данных на Railway (Volume)

**Контекст.** Пути данных захардкожены относительно корня репозитория:
`server.py:56-59` и `cli.py:18-20` (`ROOT_DIR = Path(__file__).parent`;
`SESSIONS_DIR/OUT_DIR/DATA_DIR = ROOT_DIR / ...`). Railway Volume монтируется в
ОДНУ директорию, поэтому просто «подложить» три папки нельзя.

**Что сделать.**
- Ввести env `ENGLISH_TUTOR_DATA_ROOT`: если задан, `SESSIONS_DIR`, `DATA_DIR`
  и `OUT_DIR` (или как минимум `out/sessions` + `out/history.json`) живут под
  ним. Определить в одном месте (cli.py) и импортировать в server.py, чтобы не
  разъехалось.
- При старте сервера, если `ENGLISH_TUTOR_DATA_ROOT` задан и пуст — засеять его
  копией версионированных `sessions/`, `data/`, `out/` из репозитория
  (однократная миграция, идемпотентно: не перезаписывать существующее).
- Нюанс `out/web`: статику генерирует билд и она нужна и локально, и на Vercel.
  Проще всего: `out/web` остаётся в repo-пути (эфемерно на Railway — не страшно,
  пересоздаётся при старте), а под DATA_ROOT уходят только изменяемые данные:
  `sessions/`, `data/`, `out/sessions/`, `out/history.json`.
- На Railway: создать Volume, смонтировать (например в `/data`), задать
  `ENGLISH_TUTOR_DATA_ROOT=/data`.

**Критерий готовности.** Залить тестовую сессию через API на Railway →
редеплой → сессия и её analysis.json на месте; локальный запуск без env
работает по-старому.

### 1.2 Связать Vercel-фронт с Railway API

**Контекст.** Фронт уже умеет внешний api_base: `web/common.js` резолвит
приоритетом `?api_base=` из URL → inline `window.ENGLISH_TUTOR_API_BASE_URL`
(инжектится билдом из env `ENGLISH_TUTOR_API_BASE_URL`, см. `write_web_assets`
в cli.py) → localStorage. `loadHistory`/`loadAnalysis` ходят через `apiUrl()`.
Сейчас в закоммиченном `out/web` инжектнута пустая строка — страницы на Vercel
читают только статические JSON-копии из самого `out/web` (устаревают до
следующего пуша).

**Что сделать.**
- Собрать `out/web` с `ENGLISH_TUTOR_API_BASE_URL=<https://…railway.app>` и
  закоммитить (Vercel подхватит автоматически). Либо, если не хочется хранить
  URL в артефакте, — задокументировать одноразовый заход по
  `https://<vercel>/highlights.html?api_base=<railway>` на каждом из двух
  компьютеров (localStorage запомнит). Первый вариант надёжнее: работает в любом
  браузере/инкогнито без ритуала.
- На Railway задать `ENGLISH_TUTOR_CORS_ORIGIN=https://<vercel-домен>` —
  сервер уже поддерживает (см. CORS-обработку в server.py).
- Проверить все вызовы: history, analysis, upload-audio, reanalyze,
  rebuild-annotations, delete, highlight-exercise — все должны идти через
  `ET.apiUrl`/API_BASE (в record.js свой `API_BASE` — при случае мигрировать на
  `window.ET`, но это не блокер).

**Критерий готовности.** С двух разных компьютеров открытые Vercel-страницы
показывают данные с Railway; запись, сделанная с одного, через ~2 минуты видна
на другом без git-пуша.

### 1.3 Минимальный токен доступа к write-API

**Контекст.** Бэкенд без аутентификации; на Railway с публичным URL это значит:
кто угодно может заливать аудио (тратить AssemblyAI/OpenAI кредиты), удалять
сессии, дёргать генерацию упражнений. Полная приватность (Cloudflare Access на
2 email) осознанно отложена (см. docs/adr/0004), но живой write-API с чужими
ключами без замка оставлять нельзя.

**Что сделать.**
- Env `ENGLISH_TUTOR_TOKEN` на бэкенде. Если задан — все POST `/api/*`
  требуют заголовок `X-ET-Token` (или `?token=`), иначе 401. GET-чтение JSON
  оставить открытым (приватность контента — отдельное отложенное решение).
- На фронте: при первом 401 спросить токен через `prompt()`, хранить в
  localStorage, подставлять в запросы через общий хелпер в `common.js`
  (расширить `ET.apiUrl` или добавить `ET.apiFetch`).
- Это защита от случайных сканеров, не криптография — зафиксировать это
  ограничение в docs/operations.md.

**Критерий готовности.** POST без токена → 401; обе машины после одноразового
ввода токена работают как раньше; локальный dev без env — без токена.

### 1.4 Бэкап данных Railway → git

**Контекст.** После 1.1 канонические данные живут на Railway Volume, а не в
git. Git остаётся бэкапом и историей (ADR-0003 требует версионирования сессий).
Нужен путь возврата данных в репозиторий.

**Что сделать.**
- Эндпоинт `GET /api/export` (с токеном из 1.3): zip/tar с `sessions/`,
  `data/`, `out/sessions/`, `out/history.json`.
- Локальный скрипт `sync_from_prod.py`: скачать экспорт, разложить в рабочую
  копию, показать diff; коммит руками. Периодичность — после каждой сессии или
  раз в несколько недель, на усмотрение.

**Критерий готовности.** Один запуск скрипта приводит локальный репозиторий в
соответствие с Railway; свежая сессия появляется в git-истории.

### 1.5 Убрать трение со страницы записи

**Контекст.** `web/record.html` + `web/record.js`: сейчас поля
recorder/other/topic/date. Участники навсегда Roman/Andrey — поля имён это
еженедельный бессмысленный ввод. `buildQuery()` в record.js собирает параметры
для `/api/upload-audio`.

**Что сделать.**
- Захардкодить participants: записывающий = Roman (recorder), партнёр = Andrey.
  Поля имён убрать из формы (в крайнем случае — спрятать в «advanced»).
- Topic сделать опциональным (дефолт по дате), дата уже автозаполняется.
- Идеал: на странице одна крупная кнопка Start, статус-таймлайн (уже есть
  upload progress timeline), после завершения — ссылка на разбор сессии.

**Критерий готовности.** Полный цикл записи = 2 клика (Start → Stop) плюс
браузерный диалог шаринга экрана; никакого набора текста.

## Этап 2 — учебная петля (North Star)

### 2.1 Стабилизировать LLM-извлечение ошибок

**Контекст.** Все плотности грамматики (`participants[].derived.grammar`)
считаются из сохранённых `annotation_items` (см. `finalize_derived_metrics` в
cli.py и docs/metrics-and-taxonomy.md — таксономия 7 категорий: TENSE, VERB,
ARTICLE, PREP, ORDER, WORD, COLLOC). Аннотации делает chunked-флоу
(OPENAI_ANNOTATION_MODEL, gpt-5-mini через /v1/responses; нюансы в конце
tasks.md — temperature НЕ поддерживается gpt-5-mini, управлять
reasoning.effort). Проблема: разброс от запуска к запуску — одинаковый
транскрипт даёт разное число находок, тренды шумят.

**Что сделать.**
- Промпт аннотации: жёстко зафиксировать таксономию (перечислить 7 кодов с
  определениями и примерами прямо в промпте), потребовать `category` из списка
  в каждом item, запретить прочие категории.
- Минимизировать вариативность: фиксированный порядок/размер чанков,
  детерминированные параметры где доступны.
- Замерить: прогнать одну и ту же сессию 3 раза, сравнить количество и
  категории находок до/после. Цель — расхождение по category counts в разы
  меньше текущего.
- Если формула/категории поменяются — поднять `analysis_version.taxonomy`, ADR,
  `python cli.py --recompute-derived` (процедура в docs/metrics-and-taxonomy.md
  «Change procedure»).

**Критерий готовности.** Повторный прогон той же сессии даёт стабильные
per-category counts; все items имеют категорию из v1-списка.

### 2.2 Фокус-ошибки и статус «закрыто»

**Контекст.** Сердце North Star. Сейчас прогресс — это графики
(web/progress.js), которые надо интерпретировать самому. Нужна механика:
выбрали 2–3 фокуса → следующая сессия явно отвечает, закрыты ли.

**Что сделать.**
- Данные: `data/focus.json` — список записей
  `{participant, category_code, note, examples[], status: active|closed,
  set_date, closed_date}`. Живёт в DATA_DIR (попадает под Volume и бэкап).
- API: `POST /api/focus` (create/update/close, с токеном) + отдача focus.json
  через GET.
- UI на странице Session (highlights.html/highlights.js): у каждой категории в
  разборе кнопка «В фокус»; блок «Фокус недели» вверху страницы.
- Отчёт о закрытии: при анализе новой сессии сравнить плотность фокус-категории
  с сессией, где фокус поставлен (данные уже есть в
  `derived.grammar.by_category_density` в history). Правило v1, простое:
  плотность упала ≥40% и в сессии ≥120 англ. слов (порог low-sample уже принят
  в UI) → предложить закрыть. Показывать «ARTICLE: 0.86 → 0.46 ✓ можно
  закрывать» на Session и в брифинге (2.3).
- Закрытые фокусы копятся в список побед — показывать на Progress.

**Критерий готовности.** Полный цикл на реальных данных: поставить фокус →
записать сессию → увидеть вердикт по фокусу → закрыть; закрытый виден в истории.

### 2.3 Брифинг «This week» — главная страница

**Контекст.** Запланированный Home (сейчас лендинг — Session). Смысл: перед
звонком каждый открывает сайт и за 2 минуты знает, над чем работать. Генерация
должна быть серверной/билдовой, чтобы страница была статичной и мгновенной.

**Что сделать.**
- Генератор `build_briefing()` в cli.py: на входе history + последние
  analysis + focus.json; на выходе `out/briefing.json`:
  - по каждому участнику: активные фокусы со статусом-динамикой, топ-3
    категории по средней плотности за последние 3 сессии, 2–4 живых примера
    ошибка→исправление из последних annotation_items;
  - тренд-снэпшот: 2–3 метрики (error density, l1_fallback_pct, MATTR)
    «стало лучше/хуже» стрелками, без графиков.
- Вызывать генератор в конце анализа каждой сессии и при reanalyze.
- Страница `web/home.html` + `home.js`: рендерит briefing.json (через
  `ET.apiUrl`), двумя колонками Roman | Andrey. Добавить в NAV (common.js)
  первым пунктом «This week», сделать её редиректом с `/`.
- LLM для брифинга НЕ обязателен в v1 — всё собирается из готовых данных.
  LLM-совет («сформулируй фокус недели одной фразой») — опциональное улучшение.

**Критерий готовности.** После анализа сессии открытие сайта с любого
компьютера показывает актуальный брифинг без ручных действий.

### 2.4 Упражнения в ритуал

**Контекст.** `POST /api/highlight-exercise` (server.py:358) уже генерирует
упражнение по категории+примерам (OPENAI_EXERCISE_MODEL, дефолт gpt-4o-mini),
но живёт кнопкой в разборе — вне ритуала, брат сам не догадается.

**Что сделать.**
- При генерации брифинга (2.3) авто-генерировать 3–5 упражнений на человека по
  его активным фокус-категориям (переиспользовать
  `call_openai_highlight_exercise`), складывать в briefing.json.
- Показывать в брифинге в раскрывающемся блоке: задание → ответ по клику.
- Кнопку ручной генерации в разборе оставить как есть.

**Критерий готовности.** В брифинге у каждого есть свежие упражнения по его
фокусам, доступные с любого компьютера без OpenAI-ключа на клиенте.

## Этап 3 — потом (в порядке появления желания)

- **Telegram-доставка** брифинга и результата разбора (решение: пока сайт ок;
  бот на stdlib + два chat_id, cron на Railway).
- **Приватность по-настоящему**: Cloudflare Access на 2 email перед Vercel и
  Railway (закрывает и публично доступные транскрипты — известный принятый
  риск, ADR-0004).
- **Косметика**: rename highlights.html→session.html, миграция record.js на
  `window.ET`, dark mode, mobile-вёрстка.
- **Метрики роста**: vocabulary growth (новые слова за сессию против
  накопленного словаря) — единственная напрашивающаяся новая метрика; поднимет
  `analysis_version.metrics`.

## Правила работы (для любой сессии по этому плану)

- Фронт редактировать ТОЛЬКО в `web/`; `out/web` генерируется
  (`write_web_assets` в cli.py) и коммитится.
- Стек: Python 3.13, только стандартная библиотека (urllib, http.server) — не
  добавлять зависимости без явного решения.
- Изменение формул метрик/таксономии → version bump + ADR + recompute
  (docs/metrics-and-taxonomy.md, «Change procedure»).
- После задач этапа 1 боевые данные живут на Railway Volume; локальная копия и
  git — через `sync_from_prod.py` (задача 1.4).
