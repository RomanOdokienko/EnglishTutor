# План развития — English Tutor

Это активный roadmap (июль 2026). Исторический лог MVP — [tasks.md](tasks.md),
архитектура — [Architecture.md](Architecture.md), справочники — [docs/](docs/README.md).
Задачи рассчитаны на итеративную работу: каждую можно взять в отдельной сессии,
контекст внутри задачи самодостаточен.

## Куда движемся (North Star)

Инструмент навсегда для двоих (Roman + Andrey), еженедельные удалённые звонки.
Цель — не «статистика после звонка», а замкнутая учебная петля:

> **план перед звонком → звонок с записью в одну кнопку → разбор → конкретные
> ошибки помечены как фокус → на следующем звонке видно, что закрыто.**

Ретроспективная половина построена (запись → AssemblyAI multichannel →
детерминированные derived-метрики v1 → страницы Session/Progress). Не построена
проактивная половина: брифинг, фокус-ошибки, закрытие. Это этап 2.

Сознательно отложено: Telegram-доставка (пока ок заходить на сайт), dark mode,
mobile, переименование highlights.html→session.html, полноценная приватность
(Cloudflare Access) — не брать эти задачи, пока не закрыты этапы 1–2.

## Целевая топология (решение принято, июль 2026)

Три среды, у каждой своя роль:

| Среда | Роль |
| --- | --- |
| **Railway** (уже задеплоен, авто-деплой) | Живой бэкенд для ОБОИХ пользователей: приём записи, транскрибация, анализ, хранение sessions/data/out, отдача свежих JSON. Ключи OpenAI/AssemblyAI живут только здесь. |
| **Vercel** (уже задеплоен, авто-деплой из git) | UI: статика из `out/web`. Страницы ходят за данными и API на Railway (api_base). |
| **Локалка Романа** | Только разработка. Не участвует в боевом сценарии. |

Боевой сценарий: Роман открывает Vercel-страницу Record во время звонка →
кнопка → стоп → браузер шлёт аудио напрямую на Railway → Railway транскрибирует
и анализирует → обе страницы (у Романа и у брата) читают свежие JSON с Railway.
Git-пуш для появления данных у брата НЕ нужен (git остаётся источником кода и
бэкапом данных, см. задачу 1.4).

Ключевой известный риск Railway: **файловая система эфемерна** — всё, что
сервер записал (sessions/, out/, data/), исчезает при редеплое, если не
подключён Volume. Пока задача 1.1 не сделана, боевые записи на Railway делать
нельзя.

## Этап 1 — боевая петля: «нажал запись → статистика у обоих»

### 1.1 Персистентное хранилище данных на Railway (Volume)

**Контекст.** Пути данных захардкожены относительно корня репозитория:
`server.py:56-59` и `cli.py:18-20` (`ROOT_DIR = Path(__file__).parent`;
`SESSIONS_DIR/OUT_DIR/DATA_DIR = ROOT_DIR / ...`). Railway Volume монтируется в
ОДНУ директорию, поэтому просто «подложить» три папки нельзя.

**Что сделать.**
- Ввести env `ENGLISH_TUTOR_DATA_ROOT`: если задан, `SESSIONS_DIR`, `DATA_DIR`
  и `OUT_DIR` (или как минимум `out/sessions` + `out/history.json`) живут под
  ним. Определить в одном месте (cli.py) и импортировать в server.py, чтобы не
  разъехалось.
- При старте сервера, если `ENGLISH_TUTOR_DATA_ROOT` задан и пуст — засеять его
  копией версионированных `sessions/`, `data/`, `out/` из репозитория
  (однократная миграция, идемпотентно: не перезаписывать существующее).
- Нюанс `out/web`: статику генерирует билд и она нужна и локально, и на Vercel.
  Проще всего: `out/web` остаётся в repo-пути (эфемерно на Railway — не страшно,
  пересоздаётся при старте), а под DATA_ROOT уходят только изменяемые данные:
  `sessions/`, `data/`, `out/sessions/`, `out/history.json`.
- На Railway: создать Volume, смонтировать (например в `/data`), задать
  `ENGLISH_TUTOR_DATA_ROOT=/data`.

**Критерий готовности.** Залить тестовую сессию через API на Railway →
редеплой → сессия и её analysis.json на месте; локальный запуск без env
работает по-старому.

### 1.2 Связать Vercel-фронт с Railway API

**Контекст.** Фронт уже умеет внешний api_base: `web/common.js` резолвит
приоритетом `?api_base=` из URL → inline `window.ENGLISH_TUTOR_API_BASE_URL`
(инжектится билдом из env `ENGLISH_TUTOR_API_BASE_URL`, см. `write_web_assets`
в cli.py) → localStorage. `loadHistory`/`loadAnalysis` ходят через `apiUrl()`.
Сейчас в закоммиченном `out/web` инжектнута пустая строка — страницы на Vercel
читают только статические JSON-копии из самого `out/web` (устаревают до
следующего пуша).

**Что сделать.**
- Собрать `out/web` с `ENGLISH_TUTOR_API_BASE_URL=<https://…railway.app>` и
  закоммитить (Vercel подхватит автоматически). Либо, если не хочется хранить
  URL в артефакте, — задокументировать одноразовый заход по
  `https://<vercel>/highlights.html?api_base=<railway>` на каждом из двух
  компьютеров (localStorage запомнит). Первый вариант надёжнее: работает в любом
  браузере/инкогнито без ритуала.
- На Railway задать `ENGLISH_TUTOR_CORS_ORIGIN=https://<vercel-домен>` —
  сервер уже поддерживает (см. CORS-обработку в server.py).
- Проверить все вызовы: history, analysis, upload-audio, reanalyze,
  rebuild-annotations, delete, highlight-exercise — все должны идти через
  `ET.apiUrl`/API_BASE (в record.js свой `API_BASE` — при случае мигрировать на
  `window.ET`, но это не блокер).

**Критерий готовности.** С двух разных компьютеров открытые Vercel-страницы
показывают данные с Railway; запись, сделанная с одного, через ~2 минуты видна
на другом без git-пуша.

### 1.3 Минимальный токен доступа к write-API

**Контекст.** Бэкенд без аутентификации; на Railway с публичным URL это значит:
кто угодно может заливать аудио (тратить AssemblyAI/OpenAI кредиты), удалять
сессии, дёргать генерацию упражнений. Полная приватность (Cloudflare Access на
2 email) осознанно отложена (см. docs/adr/0004), но живой write-API с чужими
ключами без замка оставлять нельзя.

**Что сделать.**
- Env `ENGLISH_TUTOR_TOKEN` на бэкенде. Если задан — все POST `/api/*`
  требуют заголовок `X-ET-Token` (или `?token=`), иначе 401. GET-чтение JSON
  оставить открытым (приватность контента — отдельное отложенное решение).
- На фронте: при первом 401 спросить токен через `prompt()`, хранить в
  localStorage, подставлять в запросы через общий хелпер в `common.js`
  (расширить `ET.apiUrl` или добавить `ET.apiFetch`).
- Это защита от случайных сканеров, не криптография — зафиксировать это
  ограничение в docs/operations.md.

**Критерий готовности.** POST без токена → 401; обе машины после одноразового
ввода токена работают как раньше; локальный dev без env — без токена.

**Частично сделано (июль 2026).** `ENGLISH_TUTOR_TOKEN` введён и обязателен для
`POST /api/import-session` (см. 1.4): без env — 503, с неверным токеном — 401,
сравнение constant-time. Остальные POST пока открыты (фронт не трогали —
включение токена на них требует `ET.apiFetch` c prompt/localStorage, это
оставшаяся часть задачи).

### 1.4 Бэкап данных Railway → git

**Контекст.** После 1.1 канонические данные живут на Railway Volume, а не в
git. Git остаётся бэкапом и историей (ADR-0003 требует версионирования сессий).
Нужен путь возврата данных в репозиторий.

**Что сделать.**
- Эндпоинт `GET /api/export` (с токеном из 1.3): zip/tar с `sessions/`,
  `data/`, `out/sessions/`, `out/history.json`.
- Локальный скрипт `sync_from_prod.py`: скачать экспорт, разложить в рабочую
  копию, показать diff; коммит руками. Периодичность — после каждой сессии или
  раз в несколько недель, на усмотрение.

**Критерий готовности.** Один запуск скрипта приводит локальный репозиторий в
соответствие с Railway; свежая сессия появляется в git-истории.

**Частично сделано (июль 2026) — обратное направление.** Рассинхронизация
случилась раньше, чем бэкап: на проде жила сессия 2026-07-14 и живой
focus.json, которых не было в git (втянуты руками read-only GET'ами), а
severity-переаннотация архива была сделана локально. Для доставки локальных
артефактов на прод без повторного прогона модели добавлены
`POST /api/import-session` (токен из 1.3, пишет только
meta/transcript/timings + analysis, сам пересобирает history/web) и клиент
`push_to_prod.py`. Направление прод→git (export + sync_from_prod.py) всё ещё
предстоит; правило: прод-данные канонические, git догоняет прод.

### 1.5 Убрать трение со страницы записи

**Контекст.** `web/record.html` + `web/record.js`: сейчас поля
recorder/other/topic/date. Участники навсегда Roman/Andrey — поля имён это
еженедельный бессмысленный ввод. `buildQuery()` в record.js собирает параметры
для `/api/upload-audio`.

**Что сделать.**
- Захардкодить participants: записывающий = Roman (recorder), партнёр = Andrey.
  Поля имён убрать из формы (в крайнем случае — спрятать в «advanced»).
- Topic сделать опциональным (дефолт по дате), дата уже автозаполняется.
- Идеал: на странице одна крупная кнопка Start, статус-таймлайн (уже есть
  upload progress timeline), после завершения — ссылка на разбор сессии.

**Критерий готовности.** Полный цикл записи = 2 клика (Start → Stop) плюс
браузерный диалог шаринга экрана; никакого набора текста.

## Этап 2 — учебная петля (North Star)

### 2.1 Стабилизировать LLM-извлечение ошибок

**Контекст.** Все плотности грамматики (`participants[].derived.grammar`)
считаются из сохранённых `annotation_items` (см. `finalize_derived_metrics` в
cli.py и docs/metrics-and-taxonomy.md — таксономия 7 категорий: TENSE, VERB,
ARTICLE, PREP, ORDER, WORD, COLLOC). Аннотации делает chunked-флоу
(OPENAI_ANNOTATION_MODEL, gpt-5-mini через /v1/responses; нюансы в конце
tasks.md — temperature НЕ поддерживается gpt-5-mini, управлять
reasoning.effort). Проблема: разброс от запуска к запуску — одинаковый
транскрипт даёт разное число находок, тренды шумят. Это блокер доверия к
любым цифрам на карточках Session (2.2) и к правилу закрытия фокуса (2.3) —
делать первой.

**Что сделать.**
- Промпт аннотации: жёстко зафиксировать таксономию (перечислить 7 кодов с
  определениями и примерами прямо в промпте), потребовать `category` из списка
  в каждом item, запретить прочие категории.
- Минимизировать вариативность: фиксированный порядок/размер чанков,
  детерминированные параметры где доступны.
- Замерить: прогнать одну и ту же сессию 3 раза, сравнить количество и
  категории находок до/после. Цель — расхождение по category counts в разы
  меньше текущего.
- Если формула/категории поменяются — поднять `analysis_version.taxonomy`, ADR,
  `python cli.py --recompute-derived` (процедура в docs/metrics-and-taxonomy.md
  «Change procedure»).

**Критерий готовности.** Повторный прогон той же сессии даёт стабильные
per-category counts; все items имеют категорию из v1-списка.

**Сделано (июль 2026).** Корневая причина разброса — символьные индексы в
контракте ответа: gpt-5-mini на low reasoning effort случайно возвращал пустые
списки на целые чанки (0–26 находок на одном и том же чанке). Индексы убраны из
схемы (модель цитирует фразу дословно, `locate_annotation_spans` находит
позиции детерминированно), category enum форсится через json_schema и для
gpt-5-моделей, таксономия с определениями и примерами вшита в промпт, бюджет
токенов поднят до 8000 (reasoning делит бюджет с ответом), fallback ловит и
Invalid JSON. Замер на сессии 2026-02-09 (3 полных прогона): всего находок
142/144/141, по участникам 57/57/55 и 85/87/85, 100% с категорией — против
падений и разброса 0–96 до правок. Прогон сессии ~5 минут.

### 2.2 Session-фокус: evidence-карточки вместо авто-коуча

**Контекст.** Блок «This week focus» на Session (web/highlights.js) выдаёт
выводы («Fix First», «High Priority», «biggest drag») там, где должен
показывать доказательства. Под капотом: скрытый `priorityScore` с выдуманными
константами impact/fixability по категориям (highlights.js: `scoreImpact`,
`scoreFixability`), слоты Fix First / Easy Win / Watch Next с тремя разными
формулами, шаблонные тексты по категории (`defaultFocusGuidance`,
`buildPracticeTask`, `buildNextCallTrigger` + `build_insight_reason` в cli.py) —
одни и те же каждую неделю, имитируют персональный анализ. Recurrence считается
на клиенте бинарно (категория «встречалась», неважно 1 раз или 9) по
подгруженным прошлым analysis.json, хотя плотности уже лежат в history.
Плюс параллельный источник цифр: карточки читают `llm.annotation_grammar`
(cli.py, `apply_annotation_grammar`), а не канонический `derived.grammar`.

**Решение (июль 2026).** Страница Session — свидетель, а не тренер: карточка
показывает факты и материал, выбор фокуса делает человек (2.3).

**Что сделать.**
- Карточка категории строится только из данных: count, плотность на 100 слов,
  тренд против последних 2–3 сессий (из `history.json`
  `derived.grammar.by_category_density`, без клиентского пересчёта прошлых
  analysis.json), 2–3 лучших примера с jump to transcript.
- Ранжирование прозрачное: доля ошибок × устойчивость (плотность категории
  выше порога в ≥2 из последних 3 сессий). Обоснование приоритета — одна
  строка фактов («9 случаев, 33% ошибок, третья сессия подряд»), никакой
  скрытой формулы с весами.
- Удалить: константы impact/fixability, слоты Fix First / Easy Win / Watch
  Next, шаблонные Why it matters / Focus cue / Practice this week / Next call
  trigger, лид-абзац-автокоуч (`buildLead`).
- Единый источник цифр: Session-карточки и Progress читают `derived.grammar`;
  `llm.annotation_grammar` оставить как legacy-поле только если его читает
  что-то ещё, иначе убрать чтение на фронте.
- Кнопку ручной генерации упражнения на карточке оставить как есть.

**Критерий готовности.** У каждой карточки видно, из каких чисел она собрана;
на странице нет текста, который нельзя вывести из данных сессии/истории;
одинаковые данные дают одинаковый порядок карточек.

**Сделано (июль 2026).** Блок «This Week Focus» заменён на «Error evidence»:
карточка = категория, count / доля / плотность на 100 слов, строка тренда
«Seen in N of last M comparable sessions · density 0.86 → 0.46 → 0.68»
(история из history.json `derived.grammar.by_category_density`; сравнимая
сессия = ≥120 англ. слов, порог присутствия 0.3/100w), до 3 примеров с
переходом в транскрипт, кнопка Generate exercise. Сортировка: устойчивые
категории первыми, дальше по count/density — без скрытых весов. Удалены
priorityScore, impact/fixability-константы, слоты, шаблонные тексты и
лид-автокоуч (~450 строк JS + мёртвый CSS); фронт больше не читает
`llm.annotation_grammar` и не подгружает прошлые analysis.json.

### 2.3 Фокус-ошибки и статус «закрыто»

**Контекст.** Сердце North Star. Сейчас прогресс — это графики
(web/progress.js), которые надо интерпретировать самому. Нужна механика:
выбрали 2–3 фокуса → следующая сессия явно отвечает, закрыты ли. Система
предлагает кандидатов (топ evidence-карточек из 2.2), решение принимает
человек.

**Что сделать.**
- Данные: `data/focus.json` — список записей
  `{participant, category_code, note, examples[], status: active|closed,
  set_date, closed_date}`. Живёт в DATA_DIR (попадает под Volume и бэкап).
- API: `POST /api/focus` (create/update/close, с токеном) + отдача focus.json
  через GET.
- UI на странице Session: у каждой evidence-карточки (2.2) кнопка «В фокус»;
  блок «Фокус недели» вверху страницы показывает ТОЛЬКО поставленные фокусы и
  их статус — не автоматически угаданные.
- Отчёт о закрытии: при анализе новой сессии сравнить плотность фокус-категории
  с сессией, где фокус поставлен (данные уже есть в
  `derived.grammar.by_category_density` в history). Правило v1, простое:
  плотность упала ≥40% и в сессии ≥120 англ. слов (порог low-sample уже принят
  в UI) → предложить закрыть. Показывать «ARTICLE: 0.86 → 0.46 ✓ можно
  закрывать» на Session и в брифинге (2.4).
- Методологию зафиксировать в `docs/adr/0005-focus-methodology.md`: правило
  предложения (доля × устойчивость ≥2 из 3 сессий), правило закрытия (−40% при
  ≥120 словах). Пороги — рабочие гипотезы v1; калибровать по реальным данным
  после 2.1, когда counts стабильны.
- Закрытые фокусы копятся в список побед — показывать на Progress.

**Критерий готовности.** Полный цикл на реальных данных: поставить фокус →
записать сессию → увидеть вердикт по фокусу → закрыть; закрытый виден в истории.

**Сделано (июль 2026).** `data/focus.json` + `GET/POST /api/focus`
(set/close/remove, лимит 3 активных на участника; токен добавится в 1.3 вместе
со всеми POST). На Session: кнопка «Add to focus» на evidence-карточках (чип
«In focus» у активных), блок «Focus of the week» с вердиктом
«0.86 → 0.46/100w (−47%) ✓ ready to close» — вердикт считается на лету из
history против сессии постановки. На Progress: секция «Closed focuses».
Методология в docs/adr/0005. Полный цикл проверен в браузере на реальных
данных (ARTICLE Романа: 0.86 на 26 Jan → 0.46 на 09 Feb → закрыт).

### 2.4 Брифинг «This week» — главная страница

**Контекст.** Запланированный Home (сейчас лендинг — Session). Смысл: перед
звонком каждый открывает сайт и за 2 минуты знает, над чем работать. Генерация
должна быть серверной/билдовой, чтобы страница была статичной и мгновенной.

**Что сделать.**
- Генератор `build_briefing()` в cli.py: на входе history + последние
  analysis + focus.json; на выходе `out/briefing.json`:
  - по каждому участнику: активные фокусы со статусом-динамикой, топ-3
    категории по средней плотности за последние 3 сессии, 2–4 живых примера
    ошибка→исправление из последних annotation_items;
  - тренд-снэпшот: 2–3 метрики (error density, l1_fallback_pct, MATTR)
    «стало лучше/хуже» стрелками, без графиков.
- Вызывать генератор в конце анализа каждой сессии и при reanalyze.
- Страница `web/home.html` + `home.js`: рендерит briefing.json (через
  `ET.apiUrl`), двумя колонками Roman | Andrey. Добавить в NAV (common.js)
  первым пунктом «This week», сделать её редиректом с `/`.
- LLM для брифинга НЕ обязателен в v1 — всё собирается из готовых данных.
  LLM-совет («сформулируй фокус недели одной фразой») — опциональное улучшение.

**Критерий готовности.** После анализа сессии открытие сайта с любого
компьютера показывает актуальный брифинг без ручных действий.

**Сделано (июль 2026).** `build_briefing()` собирает `out/briefing.json` из
history, сохранённых annotation items и `focus.json` без нового LLM-вызова.
Home (`web/home.html`) показывает две колонки Roman | Andrey: активные фокусы
с baseline/latest density, три повторяющихся категории, четыре живых
error→correction примера и короткий тренд трёх метрик. Файл обновляется после
анализа, derived reanalysis и изменения focus; `/` Vercel ведёт на This week.

### 2.6 Беглость: тайминги и детерминированные метрики (запрос Андрея)

**Контекст.** Ключевая метрика Андрея — беглость, но честно она не измерялась:
`compute_fluency` — словесный объём в псевдо-CEFR, LLM-скор 0–10 субъективен,
и ни один не показывается в UI. Настоящая беглость темпоральна (темп, паузы,
длина отрезков) и требует таймингов, которые AssemblyAI отдаёт, а пайплайн
выбрасывал.

**Сделано (июль 2026, ADR-0006).** Тайминги сохраняются в
`sessions/<date>/timings.json` при каждой транскрибации;
`cli.py --backfill-timings` конвертирует сохранённый сырой ответ для старых
записанных сессий. Layer A детерминированно считает speech_rate_wpm,
articulation_rate_wpm, pauses_per_min, mean_pause_sec,
mean_length_of_run_words, speaking_time_sec — аддитивно в `derived.metrics`,
без version bump (политика аддитивных метрик в ADR-0006): отсутствие ключа =
«не измерено», не ноль. Текстовые загрузки метрик не имеют — это корректно.
Проверено на реальной записи 2026-07-13. Легаси-поля fluency не удалены
(совместимость), но окончательно deprecated.

**Сделано (UI, 2026-07-17).** Секция Fluency на Progress (темп речи, паузы
хезитации — именованные линии, НЕ композитный скор) и полоса беглости в
персональном разборе записанного звонка. Всё по presence-gate: пока ни у одной
сессии нет таймингов, секция не появляется, текстовые загрузки выглядят как
раньше.

**Сделано (UI, 2026-07-18) — беглость × точность.** Секция «Speed vs accuracy»
на Progress: scatter-траектория, где X — темп (`speech_rate_wpm`), Y — плотность
ошибок (`error_density_per_100w`), одна точка = один записанный звонок, точки
соединены во времени (полая — самая ранняя, залитая — последняя). Отвечает на
запрос Андрея «беглость важнее грамматики»: видно, держится/падает ли точность
по мере роста скорости, БЕЗ композитного скора (та же политика именованных
линий). Никакой новой хранимой метрики — переиспользует две существующие оси;
плотность ошибок на 100 слов — правильный знаменатель, сравнимый на любой
скорости. Presence-gate строже, чем у Fluency: точке нужны И тайминги, И ≥120
слов, И готовые аннотации, а секция появляется только когда у кого-то ≥2 таких
звонка (нужно направление). Причинность не заявляется (трудная тема поднимает
обе оси разом). Код: `trajectorySeries` + `makeScatter` в web/progress.js.

**Осталось (беглость × точность).** База готова, но данных нет: единственный
записанный звонок (2026-07-13) — тестовый клип на ~30 сек (52/49 слов), он не
проходит порог. График зажжётся после 2–3 полноценных записанных звонков →
реальный блокер тут не UI, а то, что полный ~30-мин звонок ещё не прошёл
end-to-end (см. запись-таймаут в CLAUDE.md). MLR как второй маркер автоматизации
(«фразы длиннее при том же уровне ошибок») — кандидат во вторую очередь.

**Осталось.** Калибровка `PAUSE_THRESHOLD_MS` по реальному звонку: 500 мс —
рабочая гипотеза v1 (это же сказано на method-странице и в ADR-0006). Смена
порога = смена методики: bump `TIMING_VERSION` + `--recompute-derived` по
архиву.

### 2.7 Severity: грубость ошибки как атрибут находки

**Контекст.** Ранжирование Session/брифинга частотное: редкая, но грубая
ошибка видна в карточках, но выпадает из топ-3 и из This week. Грубость —
свойство конкретного экземпляра, не категории (веса по категориям — ровно тот
паттерн, что выпилен в 2.2). Оценка найденной фразы — стабильная половина
задачи (2.1), поэтому severity от модели должна быть надёжнее, чем extraction.

**Что сделать.**
- ✅ Поле `severity` (blocking/noticeable/minor по влиянию на слушателя);
  finding-dict теперь собирается в одном месте (`build_finding`).
- ✅ Пилот на eval-сете (2026-07-16): 82%/87% — вопрос про severity не портит
  извлечение; распределение blocking 0 / noticeable 73 / minor 57, оценки при
  ручном просмотре адекватны, ортогональны confidence. Числа в eval/README.
- ✅ Включено по умолчанию (`OPENAI_ANNOTATION_SEVERITY=0` — выключатель);
  version bump не нужен — гейт подсчёта не изменился (обоснование в ADR-0007).
  Локальный архив переаннотирован. Прод: один POST /api/rebuild-annotations
  (не в poll-цикле!).
- ✅ UI (2026-07-17): карточка «Serious errors only» на Progress (плотность
  blocking + noticeable отдельной линией, не взвешенный композит);
  гарантированный слот «Most serious» на Session и в This week — верхний
  присутствующий уровень, независимо от частоты категории и от фокуса.
  Питается новыми аддитивными полями `derived.grammar.by_severity_count` и
  `serious_error_density_per_100w` + блоком `grossest` в briefing.json
  (контракты — в docs/contracts.md и docs/metrics-and-taxonomy.md).
- Перед тем как severity начнёт влиять на *счётные* метрики: разметить
  severity у REAL-находок эталона и замерить согласие с моделью числом.
  Сейчас severity влияет только на порядок и показ — счётный гейт не трогает.

**Критерий готовности.** На Session и в брифинге грубая редкая ошибка не
теряется; на Progress видна динамика серьёзных ошибок отдельно от всех.
